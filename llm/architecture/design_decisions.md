# 设计决策记录 (Design Decisions)

## 1. 文档目的
本文档记录 `hc-mes-sync-server` 为什么采取了极其“原始和直接”的某些独特的设计方法。理解这些历史和环境背景，有助于后续人员在架构大改/重构时进行客观的衡量。

## 2. 决策分析

### 2.1 为什么避免采用重量级的关系型数据库？ (选择了 SQLite WAL)
* **背景与约束**：此服务往往需要直接被部署于某一个车间或管理机房的单一低配置 CentOS/Ubuntu 服务器，甚至直接跑在某台作为转发节点的工控机上。现场很多时候没有 DBA 支持且禁止连通公网，无法通过 Docker Compose 甚至 K8S 轻易把控复杂的数据库组件启动及容灾备份情况。
* **架构选择**：通过将数据库降段至文件级应用内嵌系统 SQLite。
* **面临的挑战与补偿**：SQLite 如果并发写入会引发文件全锁导致阻塞报错 `SQLITE_BUSY`。为了解决它：1. 开启了 `db.pragma('journal_mode = WAL')`，大幅解耦并提升了并发写入瓶颈。 2. 数据入库操作本身封装在原子的 `db.transaction()` 并发处理中，减小了锁时间。这种配置足够在较长一段时间内满足每天 1~10 万条级的异步上报速率。

### 2.2 为什么鉴权仅使用固定的 `Bearer Token` 而不使用 JWT / OAuth？
* **背景与约束**：本服务的 99% 调用者为部署在同一内网（甚至是同厂房网段内）的桌面数据采集上传客户端，且属于全自动化系统到系统通信 (M2M)。
* **架构选择**：所有鉴权只需配置一次全局共享 `SYNC_SERVER_TOKEN`，在各客户端配置文件写入同样的密钥即可完成全部认证防护。
* **优点**：免除了分发 Session、设计 User 表、续约与分发等全部流程；防爆破已足以靠企业防火墙内部信任域拦截；大大减少了边缘机器的心跳校验开销。

### 2.3 为什么选用 `INSERT OR IGNORE` 作为幂等设计的核心支撑？
* **背景与约束**：客户端极有可能产生网络抖动或本地程序崩溃导致同一份记录向服务端发送数次，造成 MES 分析数据的严重叠加谬误。针对大批量的发送时，必须快速判定是否处理过。
* **架构选择**：在表建立之初，即硬性规定唯一标识索引三元组：`UNIQUE(client_id, source_file, line_number)`。插入时使用 `INSERT OR IGNORE` 让底层的 C 原生层快速过滤掉重复上报，而 Node 侧返回值为 `info.changes` 差异统计量。 
* **影响**：使得客户端发过来的 JSON “爱发几次发几次”，系统自身在无需执行庞大的“先查后写”逻辑下达成了优雅极致的幂等操作性能，最大化释放了主线程。

### 2.4 EJS 服务端渲染页面的抉择
* **背景与约束**：仅需要一个极为单一的报表列表呈现页以备开发人员对数据情况“肉眼审查”，没有建立完整独立的前端后台计划。
* **架构选择**：舍弃了 React/Vue 等框架，采取了上古但绝对有效的基于 Express 后端的 EJS View Engine 进行渲染。渲染出后数据直接嵌套进 HTML 骨架。没有任何构建依赖，避免了静态服务器搭建，进一步巩固了所谓"Zero-Config" 的服务精神。
