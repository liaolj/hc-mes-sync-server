# 技术选型分析 (Technology Choices)

## 1. 文档目的
本文档详细列出 `hc-mes-sync-server` 的核心技术栈和基础设施选型，并分析“为什么如此选择”，以便后续开发或接手项目的人员理解背后的设计考量。

## 2. 核心技术架构

### 2.1 编程语言与运行时
* **语言**: JavaScript (ES6+), 未使用 TypeScript
  * **选型理由**: 快速验证、灵活迭代能力强；在单一开发者或小型项目中，避免了类型系统的维护代价。针对这种具有胶水层、极简接收端性质的服务，JavaScript 可以做到极简的单文件服务器实现。
* **运行时**: Node.js (18 / 20+)
  * **选型理由**: 优异的异步非阻塞 I/O 能力，非常适合处理这种主要包含大量小文件/短 JSON 并发上报并入库的场景（典型的 I/O 密集型业务）。

### 2.2 Web 框架与路由
* **框架**: Express 4.x
  * **使用位置**: `server.js` 初始化整个系统，响应 HTTP Request
  * **选型理由**: 业界绝对标准，生态丰富、资料充实、非常成熟。不需要复杂的分层结构（如 NestJS 等企业级框架方案），Express 其极简的路由体系正应对当前简单直白的 5~10 个 API 需求。

### 2.3 数据库与持久化
* **数据库**: SQLite 3
* **驱动库**: `better-sqlite3` (无原生 ORM)
  * **使用位置**: `db.js`
  * **选型理由**: 
    1. **极简部署(零运维)**：无须繁琐安装 MySQL/PostgreSQL，部署只需 npm install 即可将驱动包含，数据库本质是一个本地文件（`data/sync_server.db`），适配工厂内部内网或者工控机环境的独立部署要求。
    2. **性能优越**：配合 `WAL`（Write-Ahead Logging）模式，SQLite 也能轻松支撑中小规模生产线的数据上报并发场景。同时 `better-sqlite3` 是当前 Node.js 执行 SQLite 最快的同步执行驱动库之一。
* **ORM 缺失说明**: 项目没有使用 Sequelize 或 Prisma，而是直接执行使用原生 SQL。
  * **推断**: 表单结构固化简单，直接手写 SQL 插入语句通过 `@client_id` 参数化执行（既能防止 SQL 注入），又直接在 `server.js` 使用 `db.transaction()` 完成批量插入记录，将性能损耗降到了最低。

### 2.4 视图展现与前后端通信
* **模板引擎**: EJS (Embedded JavaScript templating)
  * **使用位置**: `/views` 下通过服务端拼接 HTML 返回页面。
  * **选型理由**: 在极其强调简单、无构建流程的环境里，免去完整部署 Vue/React 带来复杂度的最好办法，实现单页应用开箱即用。

### 2.5 鉴权机制
* **认证方式**: 静态 Token (Pre-shared Key / API Key 模式)
  * **使用位置**: 请求 Header 的 `Authorization: Bearer <token>` 或 Query 参数 `?token=<token>`，通过 `authMiddleware` 校验。
  * **选型理由**: 项目属于封闭内网/工业场景对接，没有面向公众的防暴破需求。设备端数量有限，共享固定的 TOKEN 对于设备端接入鉴权代价最小，满足基础数据隔离安全诉求。

### 2.6 部署环境与流程管控
* **进程管理**: PM2
  * **选型理由**: Node.js 服务在生产环境中需要日志分割、崩溃自启与开机启动守护能力。`pm2` 为行业标准，极大弥补 Node.js 单线程应用异常崩溃的安全隐患。
* **CI/CD 工具**: GitHub Actions (部署脚本配置于 `.github/workflows/deploy.yml`)
  * **选型理由**: 通过 Actions 免密 SSH 登录至目标服务器并调用 `deploy.sh`，实现开发提交代码后，生产服务器自动拔取最新代码更新与应用重启，达到了代码即发布的流水线效果。

## 3. 技术债务与选型局限预警
* 目前 SQLite 在海量数据长期积累后的慢查问题或将出现，随着生产时间变长，平坦度和热阻记录表规模超过百万、千万级别时，查询将会缓慢甚至造成 WAL 文件过大。未来应考虑引入数据清理归档机制，或迁移至独立的 MySQL 服务。
* 代码中原生 SQL 与路由代码硬耦合（Controller 层未分离），若 API 继续剧增，将会极大降低 `server.js` 的可读性。
