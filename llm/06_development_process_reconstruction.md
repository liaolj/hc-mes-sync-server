# 开发过程复原 (Development Process Reconstruction)

## 1. 文档目的
本文档试图基于现有代码和脚本的逻辑，逆向推理并复原 `hc-mes-sync-server` 这个项目的演进过程。这有助于后续开发者理解“为什么代码长现在这样”。

## 2. 演化推演阶段

### **[推断] 阶段 1：点对点功能的初创期**
* **需求**：最开始很可能只有一个业务诉求——接收“平坦度”或“热阻”测试设备的检测结果并保存。
* **动作**：
  * 初始化了一个最简单的 Node.js 项目。
  * 引入 `better-sqlite3` 创建 `data/sync_server.db` 文件。
  * 书写了第一张表（比如 `flatness_records`）和第一个上传接口 `/api/upload/flatness`，只负责插入数据库。

### **[推断] 阶段 2：多业务的并线与 API 模式的固化**
* **需求**：随着第一台测试工序的成功联调，另一工序（热阻）的工控机也需要纳入集中管控。
* **动作**：
  * 在 `db.js` 扩充了 `thermal_records` 表。
  * 复制了平坦度的接口逻辑，创建了 `/api/upload/thermal`，由此形成了当前的“多路由双轨道并线”架构。
  * 制定了通用的 Token 认证机制放在请求头或 URL Query 中（`authMiddleware`）。

### **[推断] 阶段 3：可视化看板的引入**
* **需求**：后台数据库记录确实有了，但是测试工程师及工艺人员没有简单的方法可以查询它们。要求有一个可视的列表可以看数据到了没。
* **动作**：
  * 引入 `ejs` 渲染引擎。
  * 编写了根路由 `/` 及简单的 `GET /api/records/*` 查询分类。
  * 在渲染逻辑中引入了分页（Pagination）以对防页面崩溃（因为可能已有数千条检测数据）。

### **[推断] 阶段 4：现场痛点的整改与机器别名化**
* **需求**：网页虽然起来了，但上面显示所有的客户端都叫 `xx-xx-xx-xx-xx`（MAC 地址）或 `PC-00X`，完全分不清哪台是线体尾部的热阻设备；另外测试环境频繁发入假数据污染库需要清理。
* **动作**：
  * **[确凿痕迹]** 添加了 `client_aliases` 横跨关联表，并开放了 `/api/clients/:client_id/alias` 端点能够命名机器。
  * **[确凿痕迹]** 修改了 `GET /` 数据组装逻辑，并合并两个全量表的 `client_id` 来进行联合匹配展现（当前视图中联查逻辑为左连接思想）。
  * 增加了危险的 `/api/reset` 重置接口。

### **[确凿痕迹] 阶段 5：成熟化与运维脚本化**
* **需求**：每次到厂区部署服务非常痛苦，环境常常各异，人工管理进程容易在意外断电后无法自启。
* **动作**：
  * 开发了非常详尽的 `deploy.sh`。具备自动化的 Node/PM2 检测能力并自制 `.env`。
  * 接入 GitHub Actions （见 `.github/workflows/deploy.yml`）。团队已达成了云端或统一源码库到直接跳板下发厂区执行更新的代码流水线。

## 3. 历史包袱/代码痕迹说明
* **无遗留 v1/v2 目录痕迹**: 项目应该是从立项开始就小步快跑，没有出现大量的废弃文件，非常的克制。
* **存在一定的复制粘贴逻辑 [重构点]**: `server.js` 中的 `/api/upload/flatness` 与 `/api/upload/thermal` 出现了大段完全结构一致的核心插入事务 `db.transaction()` 代码，仅仅是绑定的 Key 值和表查询名称不一样。这是典型的"复制驱动开发"，暂未提取合并高阶函数进行重制。
