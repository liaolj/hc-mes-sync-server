# 风险清单与待优化项 (Risks & To-Dos)

## 1. 文档目的
汇总系统中的技术债务、当前面临的运行风险点，以及建议后续迭代优化的清单。

## 2. 运行时与架构风险

### 2.1 数据库超载隐患 (High Risk)
* **表现**: 现阶段使用单体 SQLite3 `data/sync_server.db` 无限期不断追加。
* **风险**: 两年内如果不做任何清空，SQLite 索引会变得极其庞大，导致 `/` Web 根目录的 SELECT 统计与全部计算响应变得非常缓慢，最终导致内存溢出甚至 OOM 崩溃。
* **建议(TODO)**: 加入数据历史清理策略。例如部署一个 Cron 定时任务，每隔季度将半年前的旧数据导出并 `DELETE` 清理。

### 2.2 Token 安全级别单一 (Medium Risk)
* **表现**: `SYNC_SERVER_TOKEN` 使用同一套预设字符串，散布在车间所有的工控端中。
* **风险**: 只要有一台工控机泄露该 Token，那么拿到的人可以在内网无视机器白名单随意推送虚假的质检数据甚至使用 `/api/reset` 清空全部库。
* **建议(TODO)**: 为 `/api/reset` 等高危 API 设定单独的超级管理员 Token，不再和普通设备的上传验证共享秘钥；长远应引入基于 MAC 的设备准入白名单。

## 3. 技术债务点 (Technical Debt)

### 3.1 逻辑高内聚的 `server.js`
* **表现**: `/api/upload/flatness` 与 `/api/upload/thermal` 拥有几乎长达大几十行一模一样的事务组装逻辑与 HTTP 错误返回捕捉块。
* **分析**: 这极大违背了 DRY (Don't Repeat Yourself) 原则。未来如果还需要加入“组装偏差测试仪”第三种数据，将需要拷贝第三遍。
* **修复建议 (FIXME)**: 将其抽象出一个 `Service` 层中例如 `insertBatchRecords(tableName, rows)` 函数集中负责通用校验与插入逻辑。

### 3.2 缺少完善的日志轮转机制
* **表现**: 部署仅通过 `deploy.sh` 借助 PM2 原生能力吐出标准输出 log，并没有配置 `pm2-logrotate` 等进阶插件。
* **风险**: 长期运行 `~/.pm2/logs/` 目录将被占满导致服务器磁盘 100% 引发全线瘫痪卡死。
* **修复建议**: 在 `deploy.sh` 自动安装 `pm2 install pm2-logrotate` 确保安全。
